## Issues Fixed

- Added Chrome auto-bootstrap when `globalThis.browser` is not already available.
- Made composer/mode detection more tolerant of transient ChatGPT UI and CDP delays.
- Added session-store fallback to `nodeRepl.tmpDir` when the preferred home store is not writable.
- Wait for the real `/c/...` conversation URL before recording a pending session.
- Save pending session metadata before waiting for GPT completion.
- Search fallback session store when polling or listing sessions.
- Filter assistant status labels such as `Pro thinking`, `Thinking`, `Extended Pro`, and `Thought for Ns` from returned assistant messages.
- Kept message normalization inside browser-evaluated code to avoid page-scope reference errors.
- Removed the default 512KB image guard after user feedback. Images now paste original bytes by default.
- Removed the default size block and stopped treating compression as the fix for large images.
- Added macOS native clipboard helper and `preloadedClipboard` mode for large original image paste without compression.
- Added image attachment confirmation before send.
- Changed default relay behavior to wait for ChatGPT completion rather than returning `pending`.
- Added assistant-text stability checking so long streaming answers are not collected too early.

## Not Fully Exercised

- GPT Apps such as Canva are implemented through `appName`, but were not run end-to-end to avoid creating or modifying real app artifacts during QA.
- `deep-research`, `create-image`, and project selection paths are documented and implemented, but were not fully run because they can take a long time or create side-effect artifacts.
- Durable cross-restart session persistence depends on the runtime being able to write `~/.codex/gpt-relay/sessions.json`; otherwise the helper falls back to temp storage and returns `session.stateWarning`.
- Very long GPT jobs may outlive the outer `node_repl` tool timeout. The helper now saves the pending session first; Codex should poll that session and not provide a local fallback answer.

## Remaining Runtime Dependency

Full file upload for documents and large original images depends on the user's Chrome/Codex upload permissions. In the current live environment, `fileChooser.setFiles` returned `Not allowed`, so non-image document upload correctly stopped with `ATTACHMENT_UPLOAD_PERMISSION_REQUIRED`. Small image fallback works without that upload permission, but large image/document reliability requires enabling both permission gates.
